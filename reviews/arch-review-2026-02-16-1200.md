# Architecture Review — Settler Task System — 2026-02-16

## System Overview

The settler task system is the behavioral core of the game. `SettlerTaskSystem` is the single orchestrator for all unit behavior: it drives YAML-defined job sequences (walk, work, pickup, deliver) through a fixed set of `TaskType` atoms, delegating domain-specific logic to pluggable `WorkHandler` implementations registered by domain systems (WoodcuttingSystem, StonecuttingSystem, TreeSystem). Carrier logistics are layered on top: `LogisticsDispatcher` matches resource requests to supply, picks a carrier via `CarrierSystem`, which forwards to `SettlerTaskSystem.assignCarrierJob()`. Events (`carrier:deliveryComplete`, `carrier:pickupFailed`, etc.) close the feedback loop.

## Strengths

**1. Unified execution engine.** All unit behavior flows through one system with one tick loop. This eliminates animation ownership conflicts and makes the execution order deterministic and debuggable.

**2. Data-driven job definitions.** YAML defines task sequences declaratively. Adding a new settler type (e.g., fisher) requires a YAML entry + a WorkHandler registration — no changes to the orchestrator. This is a genuine extensibility win.

**3. Clean handler abstraction.** The `WorkHandler` interface is narrow and well-designed. Domain systems only implement the callbacks they need (`findTarget`, `canWork`, `onWorkTick`, etc.). The orchestrator wraps all handler calls in try/catch with throttled logging, providing a proper error boundary between the engine and domain code.

**4. Discriminated union for job state.** `JobState = WorkerJobState | CarrierJobState` with a `type` discriminant avoids class hierarchies and gives TypeScript exhaustiveness checking at every branch point.

**5. Event-driven logistics coordination.** `SettlerTaskSystem` emits events, `LogisticsDispatcher` reacts. Neither imports the other. This is textbook loose coupling and it works well.

**6. Reservation system for inventory.** `InventoryReservationManager` prevents carrier race conditions (two carriers targeting the same output slot). Reservations are released on pickup, failure, carrier removal, and building destruction — covering all edge cases.

## Critical Issues

### 1. Dual State: CarrierState.currentJob vs. SettlerTaskSystem.JobState

**Impact:** Two independent representations of "what the carrier is doing" can drift out of sync, creating hard-to-debug phantom state.

**Current state:** When a carrier gets a delivery job, state is written to two places:
- `CarrierManager.assignJob()` sets `carrier.currentJob` (a `CarrierJob` on the entity) — `carrier-manager.ts:224-237`
- `SettlerTaskSystem.assignCarrierJob()` creates a `CarrierJobState` in `UnitRuntime.job` — `settler-task-system.ts:320-339`

But `CarrierSystem.assignDeliveryJob()` calls `settlerTaskSystem.assignCarrierJob()` and then separately calls `carrierManager.setStatus()` — it does NOT call `carrierManager.assignJob()`. So `carrier.currentJob` is never set during the normal flow. Looking at `LogisticsDispatcher.findAvailableCarrier()` (line 313), it checks `carrier.currentJob !== null` to skip busy carriers. Since `currentJob` is never set, this guard is dead code — carrier busy-ness is determined solely by `carrier.status !== CarrierStatus.Idle`.

This means:
- `CarrierManager.assignJob()` and `completeJob()` are unused API surface
- `CarrierJob` type exists but is never populated in the normal flow
- `carrier.currentJob` is always `null` yet code checks it
- If a future change starts using `assignJob()`, it will create a second source of truth that can desync with `UnitRuntime.job`

**Recommended direction:** Eliminate `CarrierState.currentJob` and `CarrierJob`. Carrier busy-ness should be determined by one source: either `CarrierStatus` (already used) or by querying `SettlerTaskSystem.isWorking(carrierId)`. The `CarrierState` type should only hold status, fatigue, and home building.

### 2. CarrierSystem Is a Pass-Through Facade, Not a Real System

**Impact:** Adds an unnecessary indirection layer and creates a circular dependency that requires post-construction wiring (`setSettlerTaskSystem`).

**Current state:** `CarrierSystem` implements `TickSystem` but its `tick()` only does fatigue recovery (13 lines of logic). Its main purpose is `assignDeliveryJob()`, which:
1. Validates the carrier via `CarrierManager`
2. Calls `settlerTaskSystem.assignCarrierJob()` (the actual work)
3. Calls `carrierManager.setStatus()`

This creates a dependency cycle:
```
GameLoop creates: CarrierSystem → LogisticsDispatcher(carrierSystem) → SettlerTaskSystem
Then:             CarrierSystem.setSettlerTaskSystem(settlerTaskSystem)  // post-hoc wiring
```

The `setSettlerTaskSystem()` pattern (using `!` non-null assertion on an uninitialized field) is a code smell that signals the dependency graph has a cycle.

**Recommended direction:** Two options:

*Option A — LogisticsDispatcher calls SettlerTaskSystem directly:*
`LogisticsDispatcher` already knows about carrier selection. Give it a direct reference to `SettlerTaskSystem` and `CarrierManager`. Remove the `CarrierSystem.assignDeliveryJob()` facade. Move fatigue recovery into `CarrierManager` (it's a pure data operation, not a system concern).

*Option B — Make CarrierSystem a pure manager:*
Merge `CarrierSystem` into `CarrierManager`. Fatigue is a manager concern. The "system" concept should be reserved for things that run in the tick loop with meaningful logic.

Either option eliminates the circular dependency and the post-construction wiring.

### 3. SettlerTaskSystem Iterates ALL Entities Every Tick

**Impact:** O(n) over all entities when only units need processing. As entity counts grow (buildings, map objects, projectiles), this becomes waste.

**Current state:** `settler-task-system.ts:417-420`:
```typescript
for (const entity of this.gameState.entities) {
    if (entity.type === EntityType.Unit) {
        this.updateUnit(entity, dt);
    }
}
```

Additionally, lines 425-429 iterate ALL runtimes to find stale entries:
```typescript
for (const id of this.runtimes.keys()) {
    if (!this.gameState.getEntity(id)) {
        this.onEntityRemoved(id);
    }
}
```

This safety net is defensive against missed `entity:removed` events. If the event path is reliable, this scan is unnecessary overhead.

**Recommended direction:** Maintain a separate unit list in GameState (or use an entity archetype query). The stale-runtime scan should be removed once the `entity:removed` event path is proven reliable — or moved to a periodic check (every 60 ticks) rather than every tick.

### 4. Building Workers Loop Endlessly on Single WORK_ON_ENTITY

**Impact:** Building workers (smith, sawmill, miller, butcher) have a two-task job (`GO_TO_TARGET` → `WORK_ON_ENTITY`), but after completing one production cycle, the job ends and the settler returns to IDLE. Next tick, `handleIdle` searches for a workplace again, starts a new job, and walks back — even though the worker is already standing at the building.

**Current state:** The YAML for building workers:
```yaml
smith.station:
  - task: GO_TO_TARGET   # Walk to building
    anim: walk
  - task: WORK_ON_ENTITY # Produce one item
    anim: work
    duration: 5.0
```

After `WORK_ON_ENTITY` completes, `completeJob()` transitions to IDLE. The worker then goes through `handleIdle` → `findTarget` (finds same building) → starts new job with `GO_TO_TARGET` → already adjacent → immediately DONE → `WORK_ON_ENTITY` again. The `GO_TO_TARGET` resolves in one tick (distance <= 1), so it's functionally fine but architecturally wasteful.

**Recommended direction:** Consider a looping job model for stationary workers. The STAY task type exists but is currently "indefinite stay." A better model: building workers should loop their production cycle while assigned to the building, only returning to IDLE when the building is destroyed or the worker is reassigned. This could be a `loop: true` flag on the job definition, or a `WORK_ON_ENTITY` that auto-repeats.

### 5. Domain "Systems" That Have No Tick

**Impact:** Naming confusion. `WoodcuttingSystem`, `StonecuttingSystem` are classes that exist solely to register a `WorkHandler` in their constructor. They have no `tick()`, no state, and no ongoing behavior. They're stored as `public readonly` fields on `GameLoop` but never called again.

**Current state:**
```typescript
// game-loop.ts:251-252
this.woodcuttingSystem = new WoodcuttingSystem(gameState, this.treeSystem, this.settlerTaskSystem);
this.stonecuttingSystem = new StonecuttingSystem(gameState, this.settlerTaskSystem);
```

These are construction-only side effects disguised as systems. `StonecuttingSystem` is 36 lines — a class wrapper around a single function call.

**Recommended direction:** Replace these with registration functions:
```typescript
registerWoodcuttingHandler(gameState, treeSystem, taskSystem);
registerStonecuttingHandler(gameState, taskSystem);
```
Or use the `FeatureRegistry` pattern that already exists for TreeFeature. This makes the intent clear: these are handler registrations, not persistent systems.

## Recommended Restructuring

### A. Flatten the Carrier Assignment Chain

**Scope:** `CarrierSystem`, `CarrierManager`, `LogisticsDispatcher`, `SettlerTaskSystem`

**Rationale:** The current 3-hop chain (`LogisticsDispatcher → CarrierSystem → SettlerTaskSystem`) exists because `CarrierSystem` was originally the carrier behavior owner. After the unification into `SettlerTaskSystem`, the middle hop became a pass-through. Removing it simplifies the dependency graph and eliminates the circular reference.

**Target state:**
- `LogisticsDispatcher` receives `SettlerTaskSystem` + `CarrierManager` directly
- `CarrierManager` absorbs fatigue recovery (a periodic `updateFatigue(dt)` method)
- `CarrierSystem` is deleted
- `SettlerTaskSystem.assignCarrierJob()` handles status transitions via `CarrierManager`
- No more `setSettlerTaskSystem()` post-construction wiring

**Dependencies:** Can be done incrementally:
1. Move fatigue into CarrierManager
2. Wire LogisticsDispatcher to SettlerTaskSystem
3. Delete CarrierSystem

### B. Eliminate Carrier Dual State

**Scope:** `CarrierState`, `CarrierManager`, `CarrierJob` type

**Rationale:** `CarrierState.currentJob` and `CarrierJob` are vestigial from before the SettlerTaskSystem unification. They create confusion about where carrier job state lives.

**Target state:**
- `CarrierState` has: `entityId`, `homeBuilding`, `fatigue`, `status`
- No `currentJob` field, no `CarrierJob` type
- `CarrierManager.assignJob()` and `completeJob()` are removed
- Carrier availability is `status === Idle && canAcceptNewJob(fatigue)`

**Dependencies:** Depends on (A) since CarrierSystem.assignDeliveryJob currently orchestrates status + job.

### C. Register Domain Handlers via FeatureRegistry

**Scope:** `WoodcuttingSystem`, `StonecuttingSystem`, `TreeSystem.registerForesterHandler()`

**Rationale:** These are not systems — they're one-time registrations. Using the existing FeatureRegistry pattern would make them consistent with TreeFeature and make GameLoop less cluttered.

**Target state:**
- `WoodcuttingFeature`, `StonecuttingFeature` as Feature modules
- Each exports a `register()` function that calls `taskSystem.registerWorkHandler()`
- GameLoop loads them via `featureRegistry.load()` alongside TreeFeature
- No more `public readonly woodcuttingSystem` fields on GameLoop

**Dependencies:** None — can be done independently.

### D. Add Unit Index for Tick Performance

**Scope:** `GameState`, `SettlerTaskSystem`

**Rationale:** Iterating all entities to find units is O(n) waste that grows with map size.

**Target state:**
- `GameState` maintains a `Set<number>` or typed list of unit entity IDs
- Updated on entity add/remove
- `SettlerTaskSystem.tick()` iterates only units
- Remove the stale-runtime safety net (replace with periodic audit in debug mode)

**Dependencies:** None — can be done independently.

## Questions for the Team

1. **Should building workers ever leave their workplace?** Currently they cycle IDLE → walk to building → produce → IDLE → repeat. If the answer is "no, they stay forever once assigned," a looping job model would be cleaner and prevent edge cases where workers wander.

2. **Is CarrierManager.assignJob() used anywhere outside tests?** If it's purely vestigial, it can be removed as part of restructuring B. If tests depend on it, the test setup needs updating.

3. **Should the stale-runtime safety net in tick() be kept?** It protects against missed `entity:removed` events but costs O(n) per tick. Is there a known scenario where events are missed, or is this purely defensive?

4. **Are there plans for multi-carrier jobs (construction supply chains)?** The current model is 1 request → 1 carrier → 1 trip. If multi-carrier coordination is planned, the carrier assignment chain needs to be designed with that in mind before simplifying.
