# Architecture Review ŌĆö 2026-02-17

## System Overview

Settlers.ts is a browser-based remake of Settlers 4 built with TypeScript, Vue 3, and WebGL2. The architecture centers on a **Game** object that owns terrain data and delegates to **GameLoop** for simulation and **Renderer** for display. Game logic flows through a **command pipeline** (place buildings, spawn units, move, select), an **EventBus** for cross-system notifications, and registered **TickSystems** for per-frame simulation. Feature modules under `src/game/features/` encapsulate domain logic (building construction, carriers, logistics, inventory, placement, trees), while `src/game/systems/` houses simpler focused behaviors (movement, pathfinding, settler tasks). A Vue composable layer (`use-renderer.ts`, `use-map-view.ts`) bridges the game engine to the UI.

The project has strong documented design rules (`design-rules.md`, `feature-modules.md`) that define a clear layered architecture and feature module pattern. The actual codebase follows these rules partially ŌĆö the feature module pattern is well-implemented for several domains, but structural drift has created a set of architectural issues worth addressing.

## Issues

### 1. GameLoop Is a God Object / Service Locator

**Impact:** Every part of the codebase that needs any game subsystem navigates through `game.gameLoop.*`, coupling all consumers to the concrete layout of GameLoop's 15 public fields. Adding, removing, or reorganizing any manager forces changes across 6+ consuming files. The constructor is 170 lines of wiring that mixes frame-loop concerns with domain object construction.

**Current state:** `GameLoop` owns 15 public `readonly` fields (managers, systems, services) and creates all of them in its constructor. External code reaches through `game.gameLoop.X` in at least 6 files:
- `game.ts` ŌĆö unwraps `settlerTaskSystem`, `buildingStateManager`, `inventoryVisualizer`
- `game-state-persistence.ts` ŌĆö 9 direct accesses to managers for save/restore
- `use-renderer.ts` ŌĆö 7 accesses for rendering state sync
- `useLogisticsDebug.ts` ŌĆö 7 accesses for debug panels
- `selection-panel.vue` ŌĆö 5 accesses for UI display
- `use-map-view.ts` ŌĆö 3 accesses for inventory rebuild

The `Game` class does not encapsulate GameLoop's internals ŌĆö its `commandContext` getter pulls out `settlerTaskSystem` and `buildingStateManager` directly.

**Recommended direction:** Split GameLoop into two concerns: (1) a **FrameLoop** that owns only tick/render scheduling, and (2) a **GameServices** container (or composition root) that creates and wires all managers/systems. Expose needed managers through `Game` facade methods or a `GameServices` interface, eliminating the `game.gameLoop.X` navigation pattern. Consumers should depend on interfaces they need, not the entire container.

### 2. EventBus and Entity Types Create Upward Dependencies from Core to Features

**Impact:** The core layer (`event-bus.ts`, `entity.ts`) depends on feature modules, violating the documented Layer Architecture (Rule 1.1). This means features cannot be added or removed without modifying core files, and it creates a structural inversion where the foundation depends on what it should support.

**Current state:**

`event-bus.ts` (Layer 3 infrastructure) imports types from feature modules:
```
import type { BuildingState } from './features/building-construction'
import type { BuildingCleanupResult } from './features/logistics/logistics-dispatcher'
import type { RequestPriority } from './features/logistics'
```

`entity.ts` (Layer 0 data) re-exports feature types and embeds them in the Entity interface:
```
export type { BuildingState } from './features/building-construction/types'
export type { CarrierState, CarrierStatus } from './features/carriers/carrier-state'
// Entity interface has optional fields typed as feature imports
tree?: import('./systems/tree-system').TreeState
construction?: import('./features/building-construction/types').BuildingState
carrier?: import('./features/carriers/carrier-state').CarrierState
```

**Recommended direction:** For `event-bus.ts`: define event payload types as structural interfaces inline within `GameEvents`, or in a shared `game-events.ts` file at the same layer ŌĆö remove direct imports from features. For `entity.ts`: remove the feature-type re-exports and optional component fields. If features need to associate state with entities, they should own that state in their own maps (e.g., `Map<entityId, BuildingState>`) rather than hanging it off the Entity struct. This is already the pattern used by `CarrierManager`, `BuildingStateManager`, etc. ŌĆö the Entity interface fields are vestigial.

### 3. Blurred Boundary Between `systems/` and `features/`

**Impact:** The documented distinction (Rule 2.0) ŌĆö `features/` for complex business domains, `systems/` for simple focused behavior ŌĆö is not consistently followed. Several modules in `systems/` have full feature-level coupling, creating confusion about where new code should go and making the layer rules unenforceable.

**Current state:**

- `systems/settler-tasks/` imports from `features/carriers`, `features/inventory`, and `features/logistics` ŌĆö it is fully coupled to the feature layer and is effectively a feature itself.
- `features/logistics/logistics-dispatcher.ts` imports back from `systems/settler-tasks` ŌĆö creating a circular dependency at the module-group level.
- `systems/production-system.ts` depends on `features/logistics`, `features/inventory`, and `features/building-construction` ŌĆö it belongs in the feature layer.
- `systems/map-buildings.ts` depends on `features/building-construction` ŌĆö also feature-level.
- `features/trees/` is a thin wrapper that re-exports `systems/tree-system.ts` ŌĆö an artificial split.
- `systems/pathfinding/` and `systems/movement/` import `isPassable` from `features/placement` ŌĆö a function that semantically belongs in a lower layer (pure terrain query).

**Recommended direction:** Promote `settler-tasks/` and `production-system.ts` into `features/`. Move `tree-system.ts` fully into `features/trees/`. Extract pure terrain queries like `isPassable` from `features/placement` down to a Layer 0/1 module (e.g., `terrain-queries.ts`) so that pathfinding and movement don't need to import from features. Enforce the boundary with a dependency-cruiser rule: `systems/**` cannot import from `features/**`.

### 4. Module-Level Singletons Couple Engine to Vue and Global State

**Impact:** `debugStats` and `gameSettings` are Vue-reactive singletons imported directly across all architecture layers, from deep engine code (commands, game loop, renderer) to UI components. This prevents the engine from running headless (without Vue), makes commands non-deterministic, and makes unit testing require global singleton manipulation.

**Current state:**

`debugStats` (module singleton with `reactive()`) is imported in 9+ files across all layers:
- `game-loop.ts` calls `debugStats.recordTick()` and `debugStats.updateFromGameState()` every tick
- `commands/command.ts` reads `debugStats.state.selectAllUnits` to modify selection behavior
- The renderer reads `debugStats` state for overlay toggles

`gameSettings` (module singleton with `reactive()`) is imported in 14+ files:
- `game-loop.ts` reads `gameSettings.state.paused` and `gameSettings.state.gameSpeed` every frame
- `commands/command.ts` reads `gameSettings.state.placeBuildingsCompleted` ŌĆö making command behavior depend on UI settings
- `renderer/view-point.ts` reads `gameSettings.state.zoomSpeed`
- `input/input-manager.ts` directly mutates `gameSettings.state.paused`

`SoundManager` is a window-stored singleton accessed via `Game.soundManager` getter.

**Recommended direction:** Pass game-affecting settings (game speed, placement mode) as part of `CommandContext` or `TickContext` rather than reading globals. Extract the Vue `reactive()` wrapper into the UI layer ŌĆö the engine should use plain objects, and the Vue bridge should wrap them reactively at the composable level. `debugStats` recording should be injectable (or a no-op in headless mode). This is primarily about inverting the dependency: the UI observes the engine, not the engine importing UI framework primitives.

### 5. Renderer Has Direct Feature Dependencies (No RenderContext Abstraction)

**Impact:** The renderer imports directly from `features/building-construction`, `features/placement`, `systems/animation`, and `input/`, creating tight coupling between the display layer and domain logic. The documented target (Rule 6.3) ŌĆö a single `RenderContext` interface that the renderer receives ŌĆö is partially implemented but not used to decouple imports.

**Current state:**

`entity-renderer.ts` imports from 9 non-renderer modules:
- `features/building-construction` ŌĆö `getBuildingVisualState`, `BuildingConstructionPhase`, `BuildingState`
- `economy` ŌĆö `EMaterialType`
- `systems/animation` ŌĆö `getAnimatedSprite`, `getAnimatedSpriteForDirection`
- `animation/index` ŌĆö `AnimationService`
- `game-settings` ŌĆö `gameSettings`
- `input/render-state` ŌĆö `PlacementEntityType`
- `game-state` ŌĆö `UnitStateLookup`

`building-indicator-renderer.ts` imports from `features/placement` and re-exports `PlacementStatus`.

A `RenderContext` type exists in `render-context.ts` and is used by `use-renderer.ts` to pass data, but `entity-renderer.ts` still imports feature types directly for its internal logic rather than receiving all data through the context.

**Recommended direction:** Complete the `RenderContext` pattern: make all data the renderer needs arrive through the context interface, including building visual state, animation data, and placement status. The renderer should only import from Layer 0 types (entity types, coordinates, material enums) and its own `RenderContext` interface. Feature-specific computation (e.g., `getBuildingVisualState`) should happen in the composable/glue layer before data enters the renderer.

### 6. Command Pipeline Bypass in Feature Systems

**Impact:** Rule 6.2 ("All mutations through commands") is violated by several feature systems that call `state.addEntity()` directly during gameplay. This breaks the command pipeline's contract and would prevent replay/undo/multiplayer if those features are ever added.

**Current state:**

Three modules create entities outside the command pipeline during normal gameplay:
- `features/inventory/inventory-visualizer.ts` ŌĆö creates `StackedResource` entities to visualize building outputs
- `features/building-construction/spawn-units.ts` ŌĆö creates `Unit` entities when buildings complete, also directly mutates `state.tileOccupancy`
- `systems/tree-system.ts` ŌĆö creates `MapObject` entities when Foresters plant trees

Additionally, the Lua scripting API (`scripting/api/goods-api.ts`, `buildings-api.ts`, `settlers-api.ts`) calls `state.addEntity()` directly.

Map initialization (`map-objects.ts`, `map-buildings.ts`) also bypasses commands but runs before the game loop starts, which is more tolerable.

**Recommended direction:** Add commands for the entity-creation cases: `spawn_visual_resource`, `spawn_building_workers`, `plant_tree`. These can be internal/system commands not exposed to the player, but they should flow through `executeCommand` to maintain the single mutation pathway. The scripting API should also route through commands. This ensures a consistent state transition log exists for all entity lifecycle events.

### 7. Vue-to-GameState Bridge Relies on Polling Through debugStats

**Impact:** `GameState` is not reactive, so Vue components cannot observe it directly. The current solution ŌĆö `debugStats` copies game state values every frame into a `reactive()` mirror ŌĆö works but is fragile, requires `void debugStats.state.frameCount` hacks to force re-evaluation, introduces one-frame lag, and conflates debug instrumentation with the primary data flow path to the UI.

**Current state:**

The game loop calls `debugStats.updateFromGameState(this.gameState)` every tick and `debugStats.updateFromGame(game)` every frame (from the update callback). These functions copy entity counts, selected entity IDs, and other state into `debugStats.state`, which is a Vue `reactive()` object.

Vue components like `selection-panel.vue` then do:
```typescript
void debugStats.state.frameCount;  // force reactive dependency
const entityId = debugStats.state.selectedEntityId;
const entity = props.game.state.getEntity(entityId);  // direct GameState call
```

This pattern means `debugStats` is not just a debug tool ŌĆö it's the primary mechanism by which the UI knows anything changed.

**Recommended direction:** Separate the concerns: create a dedicated `GameStateReactiveProxy` (or Vue composable) that exposes the subset of game state the UI needs as reactive refs, updated from the game loop's update callback. Keep `debugStats` focused on performance instrumentation and debug overlays. This makes the data flow explicit: game state -> reactive proxy -> Vue components.

## Priority Matrix

| Issue | Severity | Effort | Recommendation |
|-------|----------|--------|----------------|
| GameLoop god object | High | Med | Split into FrameLoop + GameServices; expose via Game facade |
| Core-to-feature type dependencies | High | Med | Inline event payload types; remove Entity component fields |
| systems/ vs features/ boundary blur | Med | Med | Promote coupled systems to features; extract pure queries downward |
| Module-level singletons (debugStats, gameSettings) | Med | High | Inject settings via context; extract Vue reactivity to UI layer |
| Renderer feature dependencies | Med | Med | Complete RenderContext pattern; push computation to glue layer |
| Command pipeline bypass | Med | Low | Add internal commands for entity creation in features |
| debugStats as UI data bridge | Low | Med | Create dedicated reactive proxy; keep debugStats for instrumentation |
