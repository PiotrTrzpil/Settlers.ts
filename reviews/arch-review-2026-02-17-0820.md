# Architecture Review — 2026-02-17

## System Overview

Settlers.ts is a browser-based Settlers 4 remake structured as a layered game engine. The core is organized around a **composition root** (`GameServices`) that wires managers, systems, and features together, with mutations flowing through a **command pipeline** and inter-module communication via a typed **EventBus**. A `FeatureRegistry` with topological dependency sorting supports self-registering feature modules, though most features are still manually wired. The renderer is cleanly decoupled behind an `IRenderContext` interface, with a Vue glue layer (`use-renderer.ts`) bridging features and rendering. The architecture is documented with clear design rules and layer boundaries.

## Issues

### 1. Command Module Bypasses Feature Module Boundaries

**Impact:** Violates the "single entry point" rule (Rule 2.1) and "no imports from internal" (Rule 2.2), making the building-construction feature harder to refactor since external code depends on its internal file layout.

**Current state:** `commands/command.ts` imports directly from a non-public file:
```typescript
import { BUILDING_SPAWN_ON_COMPLETE, type SpawnContext } from '../features/building-construction/spawn-units';
```
This bypasses `features/building-construction/index.ts`. The `BUILDING_SPAWN_ON_COMPLETE` map and `SpawnContext` type are internal details of how buildings spawn units on completion — logic that should live inside the feature module, not be consumed by the command dispatcher.

Additionally, `command.ts` imports 5+ feature modules directly (building-construction, placement, settler-tasks, trees), making it a high-coupling convergence point. The command for `spawn_building_units` contains ~60 lines of unit-spawning logic that really belongs inside the building-construction feature.

**Recommended direction:** Promote only the needed exports to `building-construction/index.ts`, or better — move the `spawn_building_units` handler logic into the feature module itself. The command handler should emit an event or call a feature-owned method, not contain the spawning algorithm. This aligns with the project's "events over direct calls" principle.

### 2. Feature Registration is Incomplete — Only TreeFeature Uses FeatureRegistry

**Impact:** The `FeatureRegistry` with its dependency resolution, topological sorting, and lifecycle management exists but is underutilized. Adding a new feature still requires editing `GameServices` (14-step manual wiring), which is the exact problem the registry was designed to solve.

**Current state:** `GameServices` manually creates and wires 12+ managers/systems in a 228-line constructor:
- `CarrierManager`, `BuildingInventoryManager`, `ServiceAreaManager`, `RequestManager`, `BuildingStateManager` (managers)
- `MovementSystem`, `BuildingConstructionSystem`, `SettlerTaskSystem`, `LogisticsDispatcher`, `MaterialRequestSystem` (systems)
- Only `TreeFeature` uses `FeatureRegistry.load()` — all others are hand-wired

The `FeatureContext` interface currently provides only `gameState`, `eventBus`, and `animationService`. Features like logistics need `carrierManager`, `settlerTaskSystem`, and `inventoryManager`, which aren't available through the registry's context. This makes migration hard — features can't access each other through the registry without expanding `FeatureContext` or using `getFeature()`.

**Recommended direction:** Incrementally migrate features to use `FeatureRegistry`. Start with leaf features that have no dependents (e.g., `material-requests`, `placement`), then work inward. Features that need other features should declare them as dependencies and access them via `ctx.getFeature()`. The goal: `GameServices` becomes a thin loader that calls `featureRegistry.loadAll([...allFeatures])` and collects systems.

### 3. Module-Level Singletons Prevent Multi-Instance and Test Isolation

**Impact:** `debugStats`, `gameSettings`, and `gameViewState` are module-level singletons using Vue `reactive()`. This means: (1) shared mutable state leaks between unit tests unless manually reset, (2) multiple game instances are impossible (future multiplayer/spectator), (3) deep coupling to Vue's reactivity system from within the game engine layer.

**Current state:**
- `src/game/debug-stats.ts` — `export const debugStats = new DebugStats()` (singleton)
- `src/game/game-settings.ts` — `export const gameSettings = ...` (singleton)
- `src/game/game-view-state.ts` — `export const gameViewState = new GameViewState()` (singleton)

These are imported directly by `GameLoop` (Layer 4), `command.ts` (Layer 4), and feature modules (Layer 2). The game loop reads `gameSettings.state.paused` and `gameSettings.state.gameSpeed` every frame. Commands read `gameSettings.state.placeBuildingsCompleted`. This is framework state (Vue reactive) leaking into core engine logic.

**Recommended direction:** Make these instance-owned rather than module-level. `GameSettings` and `GameViewState` should be created by the `Game` class and passed through the service container. Systems that need settings should receive them via their config objects. Debug stats can remain a singleton since they're truly global instrumentation, but game settings and view state should be per-game-instance.

### 4. systems/index.ts Re-exports from Feature Modules (Layer Violation)

**Impact:** The `systems/` barrel file imports from `features/`, which violates the documented layer architecture (Layer 1 systems must not depend on Layer 2 features). This creates confusion about canonical import paths and makes the layer boundary unenforced.

**Current state:** `src/game/systems/index.ts` re-exports:
```typescript
export { populateMapBuildings, mapS4BuildingType } from '../features/building-construction';
export { TreeSystem } from '../features/trees';
export { MaterialRequestSystem } from '../features/material-requests';
export * from '../features/settler-tasks';
```
These re-exports exist for backward compatibility — old code imported from `systems/` before the feature module migration. But they undermine the stated layer boundaries: the systems barrel now depends on feature modules.

**Recommended direction:** Remove feature re-exports from `systems/index.ts`. Update import sites to use `features/` paths directly. This is a straightforward find-and-replace refactor. The systems barrel should only export actual system-layer code (coordinates, pathfinding, movement, spatial-search, terrain-queries).

### 5. Terrain Data Has No Clear Owner

**Impact:** Terrain arrays (`groundType`, `groundHeight`) and `mapSize` are passed through multiple layers as raw parameters — stored on `Game`, passed into `CommandContext`, injected into systems via `setTerrainData()`, and also passed to renderers. There's no single module that owns terrain as a concept, so every consumer manages its own reference.

**Current state:**
- `Game` stores `groundType`, `groundHeight`, `mapSize` as public fields
- `CommandContext` carries all three
- `GameServices.setTerrainData()` pushes them to movement and construction systems
- `RenderContext` carries them for the renderer
- `use-renderer.ts` accesses them via `game.groundType`, `game.groundHeight`, `game.mapSize`

This leads to 6+ independent copies of terrain references being passed around, and systems coupling directly to raw arrays rather than a terrain query interface.

**Recommended direction:** Introduce a `TerrainData` or `MapTerrain` value object that bundles `groundType`, `groundHeight`, and `mapSize` together with query methods (`getHeight(x, y)`, `getType(x, y)`, `isPassable(x, y)`). This object would be created once from the map loader and shared by reference. Systems and commands would accept a single `terrain` parameter instead of three separate arrays. This also provides a natural place for the terrain query functions currently split between `systems/terrain-queries.ts` and `features/placement/`.

### 6. building-lifecycle.ts is an Orphaned Coordinator

**Impact:** `BuildingLifecycle` knows about every major manager (7 dependencies) and coordinates their creation/removal sequences, yet it lives as a loose file in `features/` with no module structure. It's the only file in `features/` that isn't either infrastructure (`feature.ts`, `feature-registry.ts`) or a module directory.

**Current state:** `src/game/features/building-lifecycle.ts` is a 155-line coordinator that:
- Subscribes to `building:created` and `entity:removed` events
- Orchestrates creation: service area → inventory → building state → extra handlers
- Orchestrates removal: carrier state → building state → service area → logistics → inventory → visual stacks → extra handlers
- Takes 8 dependencies via `BuildingLifecycleConfig`

It's conceptually part of the composition root — it defines the wiring order for building creation/removal. But it lives alongside feature modules, creating ambiguity about its role.

**Recommended direction:** Either move `BuildingLifecycle` into `GameServices` (where it logically belongs as wiring infrastructure), or promote it to a proper feature module directory (`features/building-lifecycle/`) with its own `index.ts`. The former is simpler; the latter makes sense if the lifecycle coordination grows more complex.

---

### 7. Renderer Barrel Exports Too Many Concerns

**Impact:** `renderer/index.ts` exports 60+ symbols spanning core rendering, camera, layer visibility, sprite metadata, sprite caching, tint utilities, and landscape internals. This makes the renderer module's public API unclear — consumers can't easily distinguish what they should import vs. what's an internal detail.

**Current state:** The barrel mixes several distinct sub-domains:
- **Core rendering:** `Renderer`, `EntityRenderer`, `LandscapeRenderer`
- **Camera:** `ViewPoint`, `IViewPoint`
- **Render context:** `IRenderContext`, `RenderContextBuilder`, builder types
- **Sprite metadata:** `Race`, `BUILDING_ICON_INDICES`, `BUILDING_SPRITE_FRAMES`, `getUnitSpriteMap`, `SpriteMetadataRegistry`, etc. (30+ exports)
- **Sprite caching:** `getAtlasCache`, `setAtlasCache`, `clearAllCaches`, etc. (12+ exports)
- **Tint utilities:** `PLAYER_COLORS`, `TINT_SELECTED`, etc.
- **Landscape internals:** `RIVER_SLOT_PERMS`

Sprite metadata alone could be ~10K lines of data tables. It's a domain unto itself (mapping game data to sprite coordinates), not a rendering concern.

**Recommended direction:** Split the renderer barrel into focused sub-modules that consumers import from directly. Sprite metadata and sprite caching are candidates for extraction into their own modules under `renderer/` (with their own barrel files). The top-level `renderer/index.ts` would then focus on core rendering APIs: `Renderer`, `EntityRenderer`, `IRenderContext`, `ViewPoint`.

## Priority Matrix

| Issue | Severity | Effort | Recommendation |
|-------|----------|--------|----------------|
| Command bypasses feature boundaries | Med | Low | Promote exports to index.ts; move spawn logic into feature |
| Incomplete feature registration | Med | High | Incrementally migrate features to FeatureRegistry |
| Module-level singletons | Med | Med | Make gameSettings and gameViewState instance-owned |
| systems/ re-exports from features/ | Low | Low | Remove feature re-exports from systems barrel |
| Terrain data has no owner | Low | Med | Introduce TerrainData value object |
