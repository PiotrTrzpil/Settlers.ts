# Architecture Review — 2026-02-17

**Focus:** Game logic module responsibilities, APIs, and abstractions

## System Overview

The game engine follows a layered architecture with a `Game` facade, `GameServices` composition root, and `GameLoop` frame scheduler. Domain logic is split between `features/` (complex multi-file modules) and `systems/` (focused single-concern systems). An `EventBus` provides loose coupling between modules. Commands go through a centralized `executeCommand()` pipeline that validates, mutates state, and emits events. Feature modules are organized with barrel files (`index.ts`) that define public APIs.

The architecture is generally well-structured with clear separation between data types (Layer 0), algorithms (Layer 1), features (Layer 2), and I/O (Layer 5). The dependency graph has no cycles. The review below focuses on structural friction points that create coupling, hamper understandability, or make it harder to add new features.

## Issues

### 1. Dual Composition Mechanism: Manual Wiring vs FeatureRegistry

**Impact:** Two conflicting patterns for composing features create confusion about which approach to use, add bridging complexity, and make the FeatureRegistry under-utilized.

**Current state:** `GameServices` manually instantiates and wires 8 managers/systems (carriers, inventory, service-areas, logistics, settler-tasks, building-construction, movement, animation) through constructor injection, setter injection, and closures. Meanwhile, `FeatureRegistry` — a proper dependency-resolving plugin system — is only used for 2 features (trees, material-requests). To bridge the two worlds, GameServices calls `featureRegistry.registerExports()` three times to inject manually-created managers into the registry's context:

```
game-services.ts:165-169
  featureRegistry.registerExports('building-construction', { buildingStateManager });
  featureRegistry.registerExports('inventory', { inventoryManager });
  featureRegistry.registerExports('logistics', { requestManager });
```

This means FeatureRegistry exists but most features don't use it, defeating its purpose of enabling self-contained, self-registering modules.

**Recommended direction:** Choose one composition approach. Either:
- (A) **Expand FeatureRegistry** to own all features — convert carriers, inventory, logistics, service-areas, settler-tasks, and building-construction into `FeatureDefinition` modules. GameServices becomes a thin bootstrap that creates the registry, loads features, and exposes the tick systems.
- (B) **Remove FeatureRegistry** and do all wiring in GameServices — simpler if you don't need hot-swappable features. Convert TreeFeature and MaterialRequestFeature to the same manual pattern.

Option A aligns with the documented design intent in `docs/architecture/feature-modules.md` and provides a consistent extension point for future features.

### 2. GameServices is a God Module

**Impact:** GameServices has too many roles — composition root, building lifecycle orchestrator, event bridge, and feature wiring layer — making it the one file everyone must modify when adding features and a magnet for coupling.

**Current state:** `game-services.ts` (339 lines) performs four distinct responsibilities:
1. **Composition root** — creates all managers and systems (lines 107-210)
2. **Building lifecycle handler** — `handleBuildingCreated()` and `handleEntityRemoved()` contain domain logic deciding which managers need notification for which building types (lines 287-338)
3. **Event bridge** — manually bridges `inventoryManager.onChange` and `requestManager.on` into EventBus (lines 222-241)
4. **Extra handler registration** — `onBuildingCreated()`/`onEntityRemoved()` provide ad-hoc extension points (lines 258-266)

The building lifecycle logic is particularly problematic. It knows about SERVICE_AREA_BUILDINGS, `hasInventory()`, `isProductionBuilding()`, and decides the cleanup order (logistics before inventory). This is domain orchestration that belongs in a dedicated feature module or lifecycle system.

**Recommended direction:** Extract building lifecycle into its own concern. A `BuildingLifecycleFeature` (or just inline event handlers per feature) would let each feature register for `building:created` and `entity:removed` independently. Each feature knows its own cleanup needs:
- ServiceAreaManager subscribes to `building:created` and creates service areas for hub types
- InventoryManager subscribes to `building:created` and creates inventories
- LogisticsDispatcher subscribes to `entity:removed` and handles its own cleanup
- etc.

This leaves GameServices as a pure composition root with no domain logic. The orchestration-order concern (logistics before inventory cleanup) can be handled via event priority or a two-phase cleanup event.

### 3. CommandContext Accumulates Feature-Specific Dependencies

**Impact:** Adding a new feature that needs command integration requires modifying the core `CommandContext` interface, violating open/closed principle. Every command handler sees every feature dependency.

**Current state:** `CommandContext` (command.ts:45-57) directly references feature-specific types:

```typescript
interface CommandContext {
    state: GameState;
    terrain: TerrainData;
    eventBus: EventBus;
    settings: GameSettings;
    settlerTaskSystem?: SettlerTaskSystem;   // feature-specific
    buildingStateManager: BuildingStateManager; // feature-specific
    treeSystem?: TreeSystem;                 // feature-specific
}
```

When a new feature needs command access, this interface grows. The `?` optionality on some fields (`settlerTaskSystem`, `treeSystem`) also creates uncertainty — are these always available or not?

**Recommended direction:** Consider one of:
- **Service locator in context:** `CommandContext` has a `getService<T>(key: string): T` method. Features register their services. Commands pull what they need. Downside: loses type safety at the call site.
- **Feature-specific command handlers:** Each feature registers its own command handler (e.g., `treeFeature.registerCommands(commandRouter)`). The command router dispatches to the appropriate feature handler without a monolithic context. This aligns with Rule 2.3 (Registration Over Import).
- **Minimal improvement:** At least make all feature references required (not optional) since they're always present after initialization, and document that CommandContext is the integration seam.

### 4. GameState Contains Entity-Type-Specific Branching

**Impact:** GameState should be a generic entity store, but it branches on entity type in `addEntity()` and `removeEntity()`, coupling it to specific subsystems (movement, resources, buildings, map objects).

**Current state:** `GameState.addEntity()` (game-state.ts:178-266) has four `if (type === EntityType.X)` branches:
- **Building:** computes footprint, sets multi-tile occupancy, emits `building:created`
- **Unit:** looks up speed, creates a MovementController
- **MapObject:** emits `mapObject:created`
- **StackedResource:** calls `resources.createState()`

Similarly, `removeEntity()` calls `movement.removeController()`, `resources.removeState()`, and emits `entity:removed`.

GameState also holds a `movement!: MovementSystem` reference set via `setMovementSystem()` — a bidirectional dependency (GameState depends on MovementSystem, MovementSystem is set on GameState).

**Recommended direction:** Make GameState a pure entity store and move type-specific logic to event handlers:
- GameState.addEntity/removeEntity only manages the entity list, ID map, and occupancy. Emits generic `entity:created` and `entity:removed` events.
- MovementSystem subscribes to `entity:created` (type=Unit) and creates controllers.
- BuildingStateManager subscribes to `entity:created` (type=Building) and handles footprint.
- etc.

This eliminates the bidirectional GameState ↔ MovementSystem coupling and makes GameState a clean Layer 3 state container.

### 5. Logistics Feature Bypasses Service-Areas Barrel File

**Impact:** Minor violation of the barrel file pattern, creating fragile coupling to internal file paths. If service-areas reorganizes its files, logistics breaks.

**Current state:** Four files in `features/logistics/` import directly from `../service-areas/service-area-queries` and `../service-areas/service-area-manager` instead of from `../service-areas`:

```
logistics-dispatcher.ts:22    → ../service-areas/service-area-queries
fulfillment-matcher.ts:10-11  → ../service-areas/service-area-manager, service-area-queries
fulfillment-diagnostics.ts:13-14 → ../service-areas/service-area-manager, service-area-queries
resource-supply.ts:10-11      → ../service-areas/service-area-manager, service-area-queries
```

These functions and types ARE exported from `service-areas/index.ts`, so the barrel file is set up correctly — the consumers just don't use it.

**Recommended direction:** Update the imports in the 4 logistics files to go through `../service-areas`. This is a simple mechanical fix.

### 6. Event Payload Types Are Duplicated, Not Shared

**Impact:** Event payload shapes for building events are hand-copied inline in `event-bus.ts` instead of referencing the `BuildingState` type from building-construction, creating drift risk.

**Current state:** `event-bus.ts` defines `building:completed` and `building:removed` payloads with inline object types that mirror `BuildingState`:

```typescript
// event-bus.ts:29-41
'building:completed': {
    entityId: number;
    buildingState: {
        entityId: number;
        buildingType: BuildingType;
        phase: number;          // should be BuildingConstructionPhase
        phaseProgress: number;
        totalDuration: number;
        elapsedTime: number;
        tileX: number;
        tileY: number;
        originalTerrain: unknown; // typed as unknown instead of the actual type
        terrainModified: boolean;
    };
};
```

The `phase` is typed as `number` instead of `BuildingConstructionPhase`, and `originalTerrain` is `unknown` instead of `ConstructionSiteOriginalTerrain | null`. This weakens type safety for event consumers.

**Recommended direction:** Import and use the actual `BuildingState` type from building-construction in the event payload. The design-rules.md Rule 3.3 says events should include sufficient context — using the real type ensures this context is well-typed. The concern about layer violations (event-bus importing from features) can be addressed by having the feature define the event payload type and the event-bus importing it as a type-only import.

### 7. Inconsistent Module Initialization Patterns

**Impact:** Three different initialization patterns coexist, creating a fragile ordering dependency that isn't enforced by the type system. A new developer can easily wire things in the wrong order.

**Current state:**
- **Constructor injection:** Most managers (CarrierManager, BuildingStateManager, MovementSystem, LogisticsDispatcher)
- **Setter injection:** `CarrierManager.setServiceAreaManager()`, `GameState.setMovementSystem()`, `TreeSystem.setCommandExecutor()`, `MovementSystem.setTileOccupancy()`, `MovementSystem.setTerrainData()`
- **Factory/registry pattern:** TreeFeature and MaterialRequestFeature via FeatureRegistry
- **Closure injection:** `getInventoryVisualizer: () => this.inventoryVisualizer` in SettlerTaskSystem config

Setter injection exists because of circular dependencies at construction time — but those circular dependencies are themselves a symptom of Issue #4 (GameState knowing about subsystems) and Issue #1 (dual composition).

**Recommended direction:** After addressing Issues #1 and #4, the setter injection cases should disappear. If MovementSystem doesn't need to be set on GameState, and all features go through FeatureRegistry, the initialization order is handled by the registry's topological sort. Failing that, document the required initialization order in GameServices with numbered steps and assertions.

## Priority Matrix

| Issue | Severity | Effort | Recommendation |
|-------|----------|--------|----------------|
| 1. Dual composition mechanism | High | High | Migrate all features to FeatureRegistry |
| 2. GameServices god module | High | Med | Extract building lifecycle to event handlers |
| 3. CommandContext accumulates deps | Med | Med | Feature-specific command handler registration |
| 4. GameState type-specific branching | Med | High | Make GameState a pure entity store |
| 5. Logistics bypasses barrel file | Low | Low | Fix 4 import paths |
| 6. Event payload type duplication | Low | Low | Import real types in event-bus.ts |
| 7. Inconsistent initialization | Med | Low | Resolves naturally from #1 and #4 |

**Recommended sequence:** #5 → #6 → #2 → #1 → #3/#4/#7. Start with the quick wins (5, 6), then extract lifecycle logic (2) which is a prerequisite for expanding the feature registry (1). Issues 3, 4, and 7 become much simpler once the composition mechanism is unified.
