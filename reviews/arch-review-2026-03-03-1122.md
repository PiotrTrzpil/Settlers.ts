# Architecture Review ‚Äî 2026-03-03

## System Overview

Settlers.ts is a browser-based Settlers 4 remake built around a layered ECS-adjacent design. A central `GameState` owns entities; `GameServices` is the composition root that creates and wires all managers and systems; `GameLoop` drives fixed-timestep ticks; a `FeatureRegistry` provides a self-registration mechanism for complex feature modules. The renderer layer is cleanly isolated behind an `IRenderContext` interface, with data marshalled by a Vue composable glue layer. The overall structure is sound and well-documented, but several inconsistencies have emerged as the codebase grew faster than the patterns could be applied uniformly.

---

## Issues

### 1. Two-Track Feature Registration: FeatureRegistry and Manual Wiring Coexist

**Impact:** Engineers must decide between two valid registration paths for every new feature, and the decision is inconsistent across existing features. `GameServices` is a ~550-line manual wiring script with explicit ordering comments (`// 1.`, `// 2.`‚Ä¶) even though `FeatureRegistry` with topological sorting was created to replace it. New features in one path can't see features from the other without breaking the pattern.

**Current state:** 9 features use `FeatureRegistry` (`trees`, `stones`, `crops`, `combat`, `inventory`, `service-areas`, `logistics`, `material-requests`, `ore-signs`). At least 7 major systems are manually wired alongside them in `GameServices`: `CarrierManager`, `BuildingConstructionSystem`, `ResidenceSpawnerSystem`, `ConstructionRequestSystem`, `LogisticsDispatcher`, `BuildingOverlayManager`, `BarracksTrainingManager`. The `carriers` feature lives in `features/` but exports no `FeatureDefinition` and is instantiated manually. `FeatureRegistry.registerExports()` exists specifically to bridge this gap, but it's only used once.

**Recommended direction:** Migrate remaining manually-wired features into `FeatureDefinition` form. `GameServices` should shrink to a thin bootstrap that calls `featureRegistry.loadAll([...])` and reads exported managers. The initialization-order comments (`// 8. MUST be registered before inventory removal`) should become declared `dependencies` in each feature definition, which the registry enforces automatically.

### 2. `CommandContext` Encodes Optional Feature References

**Impact:** Every command handler that touches a feature (trees, crops, combat, production control) uses optional chaining (`?.`) or throws a missing-context error. This means silent no-ops are possible if the context is constructed incompletely, command handlers must guard against absent deps they conceptually require, and adding a new feature to the command pipeline requires editing the shared `CommandContext` interface.

**Current state:** `CommandContext` in `commands/command.ts` has `settlerTaskSystem?`, `treeSystem?`, `cropSystem?`, `combatSystem?`, `productionControlManager?` ‚Äî all typed optional. The `game.ts` `commandContext` getter always provides all of them, but the type allows omission. Two places in `executeCommand` call `ctx.combatSystem?.releaseFromCombat(...)` silently skipping if absent. `plant_tree` and `plant_crop` throw explicitly, which is better but inconsistent.

**Recommended direction:** Either (a) make all feature references required in `CommandContext` (enforces that the context is always complete at construction) or (b) invert the dependency: commands emit rich events (`building:placed`, `tree:cut`) and features react, removing the need for feature references in `CommandContext` entirely. Option (b) is the direction the design rules already call for (Rule 6.2, Rule 2.3) and would reduce `CommandContext` to just `{ state, terrain, eventBus, settings }`.

### 3. `systems/building-overlays/` Belongs in `features/`

**Impact:** The `building-overlays` subsystem violates its own placement rule. By sitting in `systems/`, it signals "simple, focused, single-file" behavior (Rule 2.0), but it is a complex 6-file module with `BuildingOverlayManager`, `OverlayRegistry`, `TriggerSystemImpl`, overlay data loading, and event registration. It is instantiated and wired manually in `GameServices` outside `FeatureRegistry`.

**Current state:** `systems/building-overlays/` has: `building-overlay-manager.ts`, `index.ts`, `overlay-data-loader.ts`, `overlay-registry.ts`, `trigger-system.ts`, `types.ts`. `BuildingOverlayManager.registerEvents()` and `unregisterEvents()` exist, the same lifecycle pattern used by `CarrierManager`. It is directly imported and wired in `GameServices` steps 2b and 5.

**Recommended direction:** Move `building-overlays/` to `features/building-overlays/` and convert it to a `FeatureDefinition`. This would make the `systems/` vs `features/` classification accurate and allow the overlay system to self-register cleanly.

### 4. `GameLoop` Holds a Domain Dependency (`EntityVisualService`)

**Impact:** `GameLoop` is supposed to own only tick/render scheduling (per its own JSDoc). But it takes `EntityVisualService` as a constructor parameter and calls `this.visualService.update(scaledDeltaMs)` every frame inside `runAnimations()`. This is a Layer 4 module (Systems) directly holding a domain object, breaking the stated layer architecture (Rule 1.1).

**Current state:** `game-loop.ts` imports `EntityVisualService` from `./animation/entity-visual-service`, stores it as `private readonly visualService`, and has a special-cased `runAnimations()` method that sits between tick and render phases ‚Äî but unlike the render callback, it's not a registered callback, it's hardcoded.

**Recommended direction:** Convert animation updates to the same registered-callback pattern as `setRenderCallback` / `setUpdateCallback`. `GameLoop` would expose `setAnimationCallback(cb)` and know nothing about `EntityVisualService`. The glue layer (`game.ts` or `GameServices`) registers the animation update. This cleans `GameLoop` to pure scheduling with zero domain knowledge.

### 5. `GameState` Exposes Mutable Raw Collections

**Impact:** `GameState.entities` (public `Entity[]`) and `GameState.tileOccupancy` (public `Map<string, number>`) can be mutated by any caller. This bypasses the entity lifecycle (no `entity:created` event, no occupancy bookkeeping). Two known violations: `command.ts:413` manually writes to `tileOccupancy` after spawning a worker at a door tile to restore building ownership; `game-services.ts:214` passes `tileOccupancy` and `buildingOccupancy` directly to the movement system as shared mutable references.

**Current state:** Rule 5.3 says query functions should return immutable data, and Rule 6.2 says all mutations go through commands. `GameState` documents its responsibility as entity CRUD + spatial queries, but the raw collections are public. `getEntitiesInRadius()` iterates `this.entities` directly (O(n)), which is an O(n) scan for every call.

**Recommended direction:** Make `entities` readonly (`get entities(): readonly Entity[]`) and expose `tileOccupancy` only through the query methods that already exist. For the movement system, provide a narrow `IPositionUpdater` interface rather than sharing the raw Map. The direct `tileOccupancy.set` in `command.ts:413` is a workaround for a door-tile occupancy edge case ‚Äî that logic should live in `GameState` as a named method (`restoreBuildingOwnership(doorKey, buildingId)`).

### 6. `SoundManager` Is a Global Singleton

**Impact:** `SoundManager` uses the classic `getInstance()` singleton, making it impossible to test game code that interacts with audio, impossible to have multiple game instances, and creating hidden global state that contradicts the explicit dependency injection used everywhere else.

**Current state:** `game.ts` accesses it via `public get soundManager(): SoundManager { return SoundManager.getInstance(); }`. The `SoundManager` is otherwise referenced nowhere in `GameServices` or `GameLoop` ‚Äî only in `Game` itself. The singleton was likely chosen to avoid threading it through the entire dependency graph.

**Recommended direction:** Accept `SoundManager` as an optional constructor dependency of `Game` (nullable for test environments). Since `Game` is the only consumer and it already has the `GameServices` pattern for dependency injection, the singleton is unnecessary overhead. This unblocks unit tests that create `Game` instances.

### 7. Stale `.bak` Files and Vestigial Re-Export Shims

**Impact:** 13 `.bak` files pollute the source tree and confuse search results. They shadow current implementations and create false questions about which version is authoritative. Additionally, `systems/animation.ts` is a documented backward-compatibility re-export shim that should have been removed when the referencing code was updated.

**Current state:** `.bak` files present: `entity.ts.bak`, `unit-types.ts.bak`, `game-services.ts.bak`, `game-data-access.ts.bak`, `building-type.ts.bak`, `race-availability.ts.bak`, `sprite-metadata.ts.bak`, `hud-sprite-manifest.ts.bak`, `astar.ts.bak`, `index.ts.bak` (pathfinding), `use-renderer.ts.bak`, `selection-indicator.ts.bak`, `palette-data.ts.bak`. `systems/animation.ts` explicitly documents itself as a backward-compat shim and says "The functions have been moved to renderer/animation-helpers.ts".

**Recommended direction:** Delete all `.bak` files ‚Äî git history is the backup. Audit callers of `systems/animation.ts`, update them to import from `renderer/animation-helpers.ts` directly, and delete the shim. Rule 9.2 says "Delete rather than deprecate."

### 8. `refactoring-leads.md` Describes Completed Work as Outstanding Tasks

**Impact:** This document is the team's roadmap for known technical debt, but most of its tasks appear already done. New contributors reading it would think the codebase is in a much worse state than it actually is, and would not know which items still apply. The effort descriptions reference file sizes and structures that no longer exist.

**Current state:** Task 2 describes `EntityRenderer` as 1,149 lines with 29 imports ‚Äî current file is 588 lines with render passes already extracted into separate files. Task 4 describes `SettlerTaskSystem` as 870 lines with implicit state machine ‚Äî current file is 609 lines, with `UnitStateMachine`, `WorkHandlerRegistry`, `IdleAnimationController`, `WorkerTaskExecutor` all extracted. Task 6 describes `LogisticsDispatcher` as 600 lines ‚Äî now 282 lines. Task 8 (EntityCleanupRegistry) ‚Äî `entity-cleanup-registry.ts` already exists and is in active use. Task 10 (movement split) ‚Äî `blocked-state-handler.ts`, `collision-resolver.ts`, `pathfinding-service.ts` all exist. Task 11 (FeatureRegistry) ‚Äî `feature-registry.ts` exists and is used.

**Recommended direction:** Archive or rewrite `refactoring-leads.md` to reflect what was actually done and what genuinely remains. Issues 1-6 above are more accurate current leads.
