# Architecture Review — 2026-02-17

**Scope**: Integration and e2e test infrastructure. Game code flagged only where non-disruptive changes would significantly improve tests.

## System Overview

The test infrastructure has two tiers: Vitest unit tests (`tests/unit/`, 39 files) and Playwright e2e tests (`tests/e2e/`, 12 spec files). E2e tests interact with the game through 6 `window.__settlers_*__` globals, mediated by a central `GamePage` page object (1,632 lines) and a custom matcher library. Worker-scoped fixtures keep a shared browser page across tests for performance. A `WaitProfiler`, frame-relative waiting, and centralized timeout constants (`wait-config.ts`) provide deterministic synchronization. The testing guide (`docs/testing/guide.md`) is comprehensive and well-maintained.

## Issues

### 1. GamePage Is a God Object

**Impact:** The 1,632-line `GamePage` class is the single dependency of every e2e test. Any change to any concern risks breaking unrelated tests. Navigation, waiting primitives, game actions, state queries, audio controls, sprite loading, canvas diagnostics, and assertions all live in one class. A contributor adding sprite cache tests must read through movement helpers and audio controls to find the right section.

**Current state:** `tests/e2e/game-page.ts` has ~15 conceptual sections separated by comment banners. Methods range from low-level infrastructure (`_waitForFunction`, `_profiledWait`) to high-level domain actions (`placeMultipleBuildings`, `captureMovementEvents`). Every test file imports `GamePage` and accesses a different subset of its surface area.

**Recommended direction:** Extract domain-specific helpers into composition modules that `GamePage` delegates to or that tests import directly. For example:
- `game-actions.ts` — `placeBuilding`, `spawnUnit`, `moveUnit`, `placeResource`, batch variants
- `game-queries.ts` — `getGameState`, `getEntities`, `getUnitState`, `getAnimationState`, `getMovementControllerState`
- `audio-helpers.ts` — `getAudioState`, `toggleMusic`, `unlockAudio`
- `sprite-helpers.ts` — `hasSpritesLoaded`, `getLoadedUnitSprites`, `testJilLookup`, `clearSpriteCache`

`GamePage` remains the central facade for navigation/waiting/reset, and tests can import specialized modules when they need domain-specific queries. This keeps the fixture system unchanged while making each module independently readable and testable.

### 2. Commands Don't Return Created Entity IDs

**Impact:** Every action helper in `GamePage` (`placeBuilding`, `placeResource`, `spawnUnit`) contains 15-30 lines of workaround code that snapshots entity IDs before the command, executes it, then diffs the array to find the new entity. This pattern appears 4 times in `game-page.ts` (lines 829-868, 887-916, 962-995, 1292-1332). It's fragile (races if the game loop ticks between snapshot and diff), verbose, and duplicated.

**Current state:** `game.execute({ type: 'place_building', ... })` returns `{ success: true }` or `{ success: false, error: string }`. The created entity ID is never returned. Test code works around this:
```typescript
const idsBefore = new Set(game.state.entities.map((e: any) => e.id));
const cmdResult = game.execute({ type: 'place_building', ... });
const newEntity = game.state.entities.find((e: any) => !idsBefore.has(e.id) && e.type === 2);
```

**Recommended direction:** Extend `CommandResult` to include `entityId?: number` for creation commands. The command handlers (`place_building`, `spawn_unit`, `place_resource`) already have the entity reference at creation time — they just need to pass it back. This is a non-disruptive, additive change to `src/game/commands/` that would eliminate ~120 lines of workaround code in `game-page.ts` and make every action helper a simple 3-line method. It would also benefit any future game features that need to chain commands (e.g., "place building then assign worker").

### 3. View State Throttling Creates Test Latency

**Impact:** `GameViewState.updateFromGameState()` throttles entity count updates to every 500ms (`ENTITY_COUNT_INTERVAL`). Tests that assert entity counts via `toHaveBuildingCount()` or `toHaveUnitCount()` are polling against a potentially stale counter. This means: (a) tests spend up to 500ms waiting for the next count refresh, (b) the 100ms polling interval in matchers may hit stale data multiple times before succeeding, and (c) test speed is bounded by this game-side constant rather than actual game logic speed.

**Current state:** `src/game/game-view-state.ts:48-49` defines the interval. The e2e wait-config even documents this: `wait-config.ts:106` exports `DEBUG_STATS_REFRESH_MS = 500` as a known constraint. Matchers poll every 100ms, meaning up to 5 polls may return stale data before the view state refreshes.

**Recommended direction:** Add a `forceUpdate()` method to `GameViewState` that bypasses the throttle and recounts immediately. Call it from `resetGameState()` in the test infrastructure (via the debug bridge) after state mutations. Alternatively, make the interval configurable via a window global that tests can set to 0. This is a 5-line change to game code that would make count-based assertions resolve in ~100ms instead of ~500ms, saving ~400ms per assertion across ~20 tests.

### 4. Duplicated Spiral Search Algorithm

**Impact:** The "spiral from map center to find valid tile" algorithm is implemented 4 separate times in `game-page.ts` (`findBuildableTile`, `findPassableTile`, `placeMultipleBuildings`, `placeMultipleResources`) and again in `tests/unit/helpers/test-game.ts` (`findPassableTile`, `findBuildableTile`). Each implementation has slightly different validation logic (passable terrain, buildable terrain, unoccupied, etc.) but the same spiral iteration pattern. The browser-side implementations in `game-page.ts` can't be shared with the unit test helpers because they run in different contexts.

**Current state:** The 6 implementations span ~200 lines of near-identical spiral iteration code. When the terrain system changes (e.g., new terrain types), each must be updated independently. The browser-side versions also hardcode terrain type constants (`gt > 8 && gt !== 32`) that duplicate knowledge from the game engine.

**Recommended direction:** Create a shared `spiralSearch(cx, cy, w, h, predicate)` utility in the game code (`src/game/` or a shared utils module) that encapsulates the spiral iteration pattern and accepts a predicate. Export it so both browser-side `page.evaluate()` code and unit test helpers can use it (unit tests import directly, e2e tests call it via the debug bridge). The predicate-based design means each caller only specifies *what* makes a tile valid, not *how* to iterate.

### 5. Matcher Boilerplate Could Be Factored

**Impact:** All 9 custom matchers in `matchers.ts` (393 lines) follow an identical pattern: extract timeout from options, declare a `lastX` variable, call `baseExpect.poll()` with a lambda, catch errors, return pass/fail message. Each matcher is ~35 lines of which ~25 are structural boilerplate. Adding a new matcher requires copying this boilerplate and adapting 3-4 lines.

**Current state:** `tests/e2e/matchers.ts` defines 9 matchers: `toHaveEntity`, `toHaveMode`, `toHaveEntityCount`, `toHaveBuildingCount`, `toHaveUnitCount`, `toHaveUnitsMoving`, `toHaveAtLeastUnitsMoving`, `toHaveNoUnitsMoving`, `toHaveCameraAt`. The last one (`toHaveCameraAt`) has a more complex signature with tolerance, but the rest are structurally identical view-field polling matchers.

**Recommended direction:** Extract a `createViewFieldMatcher(fieldName, comparator)` factory that generates matchers from a field name and comparison function. The 7 simple matchers would become one-liners:
```typescript
toHaveUnitCount: createViewFieldMatcher('unitCount', (actual, expected) => actual === expected),
toHaveUnitsMoving: createViewFieldMatcher('unitsMoving', (actual, expected) => actual === expected),
toHaveAtLeastUnitsMoving: createViewFieldMatcher('unitsMoving', (actual, min) => actual >= min),
```
This reduces the file from ~393 lines to ~150, makes the pattern explicit, and makes adding new matchers trivial.

### 6. Debug Bridge Types Are Manually Maintained

**Impact:** `game-page.ts` defines `SettlersDebug` (lines 8-35) and `SettlersView` (lines 41-55) interfaces that mirror the game's `DebugStatsState` and `GameViewStateData`. These must be manually kept in sync with the source-of-truth types in `src/game/debug-stats.ts` and `src/game/game-view-state.ts`. If a field is renamed or added in the game code, the test types silently become stale — tests compile but access wrong/missing fields at runtime.

**Current state:** The `SettlersDebug` interface in game-page.ts is a subset of the actual `DebugStatsState` (it omits render timings, debug UI state, etc.). The `SettlersView` interface appears complete but has no compile-time link to `GameViewStateData`. There's an e2e test (`game-logic.spec.ts:143`) that verifies all window globals exist, but it can't catch property-level drift.

**Recommended direction:** Export the canonical interfaces from the game code and import them in `game-page.ts`. Since e2e tests run in Node (Playwright) not the browser, the types are only used for compile-time checking — the actual data comes from `page.evaluate()` which is untyped anyway. But importing the canonical types means the compiler would catch field renames/removals. If circular dependency concerns arise, the interfaces could live in a shared `src/game/debug-bridge-types.ts` that both game code and tests import.

### 7. @animations Tag Has No Project Binding

**Impact:** `unit-movement-animations.spec.ts` tags its describe blocks with `{ tag: '@animations' }`, but no Playwright project in `playwright.config.ts` selects or handles this tag. The tag serves only as documentation. You cannot run `npx playwright test --project=animations` to run only animation tests. Animation tests currently run in the `default` project (15s timeout) which may be too short for tests that set game speed to 1x and sample 20 animation frames.

**Current state:** Playwright config defines 5 projects: `smoke` (`@smoke`), `default` (everything else), `slow` (`@slow`), `assets` (`@requires-assets`), `visual` (`@screenshot`). The `@animations` tag is unique to `unit-movement-animations.spec.ts` and has no effect on test selection or configuration.

**Recommended direction:** Either (a) add an `animations` project with appropriate timeout (20-30s), or (b) replace `@animations` with `@slow` since the animation tests that use 1x speed are already individually tagged `@slow`. Option (b) is simpler and aligns with the existing tier system — tests that are inherently slow should use `@slow`, not a custom tag.

### 8. E2E Tests Covering Pure Logic That Could Be Unit Tests

**Impact:** Several e2e tests exercise pure game logic through the browser when they could run as unit tests at 100-500x the speed. The testing guide (docs/testing/guide.md) already identifies Tier 3 migration candidates but the migration hasn't happened systematically. Each unnecessary e2e test adds ~0.5-2s of overhead (browser IPC, frame waiting, polling) that a unit test wouldn't need.

**Current state:** Concrete candidates:
- `building-placement.spec.ts:114-130` — "building placement via game.execute() creates entity with correct attributes" — pure state verification, uses `gs` fixture but still goes through browser. Equivalent to `command.spec.ts` patterns.
- `resource-placement.spec.ts:37-53` — "resource placement via game.execute() creates entity with correct attributes" — same pattern.
- `building-placement.spec.ts:226-268` — Unit spawning tests — "spawn carrier creates entity on passable terrain", "spawned unit is on passable terrain" — pure entity + terrain checks.
- `audio.spec.ts:12-34` — "audio state toggles correctly" — state machine verification already covered more thoroughly by `music-controller.spec.ts`.

**Recommended direction:** For each candidate, check if equivalent unit test coverage exists. If not, add unit tests. Then either remove the e2e version or simplify it to a smoke check. The carrier-inventory-integration.spec.ts file already shows this pattern — it has a comment "Tier 3 — migrated from e2e" for tests that were moved. Follow the same pattern for the candidates above. Priority: building/resource placement via execute() (most straightforward), then audio toggle (already has good unit coverage).

### 9. Fixture Cleanup Pattern Is Inconsistent

**Impact:** Some test-scoped fixtures (`gp`, `gpFast`, `gs`) reset game speed in teardown, while others (`gpNormal`, `gpWithUI`, `gpWithBuilding`, `gpWithUnit`, `gpWithMovingUnit`, `gpCentered`) do not. Since all fixtures share the same worker-scoped page, this inconsistency means the page's state after a test depends on which fixture was used. Currently this works because each fixture's setup overrides the relevant state, but the asymmetric teardown pattern is confusing and fragile.

**Current state:** `fixtures.ts` — `gp` (line 243) and `gpFast` (line 369) have `await gp.setGameSpeed(1.0)` in teardown. `gpNormal` (line 261) sets speed to 1.0 in setup but has no teardown. Other fixtures don't touch speed at all.

**Recommended direction:** Adopt one consistent convention: either (a) all fixtures do cleanup in teardown, or (b) no fixtures do cleanup in teardown and all rely on setup to establish state. Option (b) is simpler and already the de facto pattern — every fixture's setup calls `resetGameState()` and sets its own speed. Remove the teardown speed resets from `gp` and `gpFast` to make the convention explicit. This eliminates dead code and prevents future confusion about whether teardown is required.
